# signal-write-allowlist.txt — Phase 5 Track A of plan-batched-signal.md.
#
# Each non-blank, non-comment line is one allowlist entry accepted by
# `forbid-signal-write.sh`. Formats (first match wins):
#
#   path                      whole-file allow (use sparingly)
#   path:receiver             allow `.write()` calls where the receiver
#                             identifier is `receiver`
#   path:line=N               pin a single line (drifts when the file is edited)
#
# Add a trailing `# rationale` comment explaining WHY — future readers
# should be able to tell whether the entry is still valid without digging
# through git history.
#
# The lint exists to prevent CLAUDE.md hang class #1 (multi-.write()
# cascade) on the migrated hot-path signals (`Signal<ChatData>`,
# `Signal<AppState>`). Those are now `BatchedSignal<T>`, whose shadow
# `.write()` is `#[deprecated] -> !`, so those sites CAN'T appear here
# even if someone tries. The entries below cover:
#   * the shadow `write` definition itself (meta — doesn't mutate anything)
#   * component-scoped `use_signal` locals (list/set state, filter toggles,
#     modal visibility) where multi-write cascades don't apply because the
#     signal has one subscriber
#   * `Signal<Locale>` (`locale_sig`) mutations — Phase 4b defers locale to
#     a later pass; low frequency, not a cascade vector.
#
# Phase 4 migrated `Signal<ChatHistoryUiState>` (`history_state`) and
# `Signal<AppSettings>` (`settings_sig`) to `BatchedSignal<T>`.
# Phase 4b migrated `Signal<ClientManager>` and `Signal<ThemeConfig>` —
# all 9 former `client_manager` allowlist entries have been removed (the
# shadow `.write()` now rejects at compile time).
# Former allowlist entries are removed; shadow `.write()` rejects them.
