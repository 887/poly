# long-read-guard-allowlist.txt — file-level and line-level exceptions for
# forbid-long-read-guard.sh (Phase 5 Track A of plan-read-guard-scoping.md).
#
# Format: path:line # reason
#   path         — relative to repo root
#   line         — 1-based line number of the `let <var> = <sig>.read();` line
#   reason       — mandatory rationale (why this is safe)
#
# Inline allowlist (preferred for single-line exceptions):
#   let x = sig.read(); // poly-lint: allow long-read-guard — <reason>
#
# CURRENTLY EMPTY BY DESIGN.
#
# 2026-05-02: the lint algorithm was upgraded to track brace depth + explicit
# `drop(<var>);` calls, which means it now natively recognises both safe
# patterns and stops false-positiving them:
#   - block-scoped read:  `let X = { let g = sig.read(); … };`
#   - explicit drop:      `let g = sig.read(); … drop(g); … sig.batch(…);`
#
# Before the upgrade the allowlist had 7 line-anchored entries that drifted
# every time someone touched the surrounding code. All 7 were removed when
# the smarter algorithm shipped — the lint stays green on the production
# tree without any of them.
#
# Add an entry here ONLY if you hit a genuinely intentional long-scoped read
# the algorithm cannot recognise (e.g. a guard intentionally held across an
# unrelated-signal write that the regex matches by coincidence). Prefer the
# inline-allowlist comment over a line entry — comments move with the code,
# line entries don't.
