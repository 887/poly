## Overview

All messenger backend clients are compiled as **WebAssembly Component Model binaries** (`.wasm`) and loaded at runtime by `crates/plugin-host`. The WIT interface (`wit/messenger-plugin.wit`) is the single source of truth.

### Why WASM plugins?

- **App Store distribution** — app ships with zero third-party client code built in
- **Sandboxing** — each plugin runs in isolation; all I/O via host-provided imports
- **Hot-swappable** — plugins updated independently of the main app
- **Community extensible** — third parties can write plugins without touching core
- **Legal isolation** — protocol-specific code is a separate artifact

## Architecture

```
   wit/messenger-plugin.wit  (WIT contract)
          │
          ├── poly-demo.wasm       (wasm32-wasip2)
          ├── poly-stoat.wasm      (wasm32-wasip2)
          ├── poly-matrix.wasm     (wasm32-wasip2)
          └── ...
                    │
            Plugin Host Runtime
            (crates/plugin-host/)
            wasmtime + Component Model
                    │
            implements Arc<dyn ClientBackend>
                    │
            ClientManager
```

## Host Imports (syscalls — host provides to plugin)

| Import | Host Implementation |
|---|---|
| `http-request` | reqwest |
| `websocket-connect` | tokio-tungstenite |
| `websocket-send` / `websocket-close` | stored connection map |
| `emit-event` | mpsc channel → event_stream() consumers |
| `storage-get/set/delete` | SQLite (namespaced per plugin+account) |
| `log` | tracing |
| `get-current-time` | std::time |

## Guest Exports (plugin provides to host)

Mirrors `ClientBackend` exactly: `authenticate`, `logout`, `get-servers`, `get-channels`, `get-messages`, `send-message`, `get-user`, `get-friends`, `get-dm-channels`, `event-stream` (via `emit-event`), `backend-type`, `backend-name`, and others.

## Event Streaming

Events cannot cross the WASM boundary as a `Stream`. Instead:

1. Guest calls `websocket-connect(url)` → host opens real WebSocket
2. Host receives WS data → calls guest's `handle-ws-data(handle, bytes)` export
3. Guest parses protocol bytes → calls `emit-event(event)` host import
4. Host forwards event to `event_stream()` consumers via mpsc channel

## Plugin Loading

| Source | Path | Use Case |
|---|---|---|
| Built-in | `include_bytes!` in binary | Demo, Poly Server (always available) |
| Local | `~/.poly/plugins/*.wasm` | User-installed |
| Downloaded | HTTPS plugin registry | Future: community marketplace |

## Platform Strategy

| Platform | WASM Runtime |
|---|---|
| Desktop (all OS) | wasmtime JIT (Cranelift) |
| Android | wasmtime JIT (aarch64 supported) |
| iOS | wasmtime AOT (`engine.precompile_component()` — JIT prohibited by iOS) |
| Web | Server-side — Axum server runs wasmtime; browser talks via HTTP relay |

## Security Model

- No filesystem access for plugins
- No raw network access — all HTTP/WS via host-mediated imports
- URL allowlisting possible at host level
- Storage isolation: per-plugin-per-account namespaced KV prefix
- Memory isolation: each WASM instance has its own linear memory
- Resource limits: fuel metering / epoch interruption

## Key Files

- `wit/messenger-plugin.wit` — WIT interface definition
- `crates/plugin-host/src/` — wasmtime host runtime
- `crates/plugin-host/src/registry.rs` — plugin discovery and registration
- `crates/plugin-host/src/host_impl.rs` — host import implementations
- `crates/plugin-host/src/storage.rs` — per-plugin SQLite KV storage
