## Storage

### Primary: SQLite (`rusqlite`)

All platforms use SQLite for persistent storage — the same `storage.sqlite3` file accessed by all shells. Plugin host uses a namespaced KV prefix per plugin+account.

### WASM / Browser

When the WASM client cannot reach the host-bridge (standalone browser without fullstack), it falls back to **IndexedDB** (`web` storage impl).

### Backup Server

The `poly-backup-server` uses SurrealDB with SurrealKV for user blob storage. This is separate from the primary SQLite storage.

## Identity Model (Session Messenger Style)

```
1. Generate Ed25519 signing keypair (ed25519-dalek)
2. Derive X25519 DH keypair from Ed25519 (curve conversion)
3. Public key (hex-encoded) = Account ID / username on backup server
4. Private key → BIP39 mnemonic = Recovery Password (export/save)
```

The identity key is generated once on first launch and stored locally. It is the user's identity on `poly-backup-server` — no email, no phone number.

## Encrypted Backup

All data synced to `poly-backup-server` is **client-side encrypted before transmission**:

1. Derive symmetric key from X25519 private key (HKDF)
2. Encrypt with AES-256-GCM (or XSalsa20-Poly1305)
3. Pad to fixed-size blocks (prevent size analysis)
4. Server stores only opaque encrypted blobs, keyed by public key

The server never sees plaintext. It only knows: public key (user ID) + encrypted blob.

## Backup Server Auth

Rate-limiting via Proof-of-Work (Anubis style):

1. Server generates random `challenge` + `difficulty` (leading zero bits)
2. Client finds `nonce` where `SHA-256(challenge || nonce)` has `difficulty` leading zeros
3. Server verifies in O(1); client work is O(2^difficulty)
4. Difficulty 20 ≈ ~1M hashes ≈ 0.5-2s on typical hardware

Server also supports a passphrase + account limit:
- Optional passphrase required to create new accounts
- Hard cap on total accounts; refuses new registrations once limit reached
- A new public key not seen before counts as "new account"

## Crates

| Crate | Use |
|---|---|
| `ed25519-dalek` | Ed25519 key generation and signing |
| `x25519-dalek` | X25519 ECDH |
| `bip39` | Mnemonic phrase encoding |
| `chacha20poly1305` / `aes-gcm` | Symmetric encryption |
| `hkdf`, `sha2` | Key derivation |
