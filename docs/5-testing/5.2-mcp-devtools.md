## Overview

Three custom Rust MCP servers manage the full app lifecycle for AI-assisted development.

**These are NOT `chrome-devtools-mcp` or any generic browser MCP.** They are custom binaries in this repo that know how to build, launch, and connect to each Poly platform shell.

| MCP Server | Binary | Platform | Debug Port |
|---|---|---|---|
| `poly-web` | `poly-web-devtools-mcp` | Chrome/Chromium | 9222 (CDP) |
| `poly-electron` | `poly-electron-devtools-mcp` | Electron | 9224 (CDP) |
| `poly-desktop` | `poly-desktop-devtools-mcp` | Wry | 9223 (HTTP eval) |

## MCP Workflow

```
launch_app
    ↓
poll get_last_build_status (every 5-10s until state != "Running")
    ↓
connect_cdp
    ↓
take_screenshot / navigate / click / evaluate_script
```

`launch_app` and `rebuild_app` are non-blocking — always poll `get_last_build_status` before interacting.

## Key Tools

| Tool | Description |
|---|---|
| `launch_app` | Start dx serve + open shell window |
| `rebuild_app` | Trigger WASM rebuild |
| `get_last_build_status` | Poll build progress (`Running`, `Success`, `Failed`) |
| `connect_cdp` | Connect to Chrome DevTools Protocol |
| `take_screenshot` | Capture current app state |
| `navigate_page` | Navigate to URL |
| `click` / `fill` / `type_text` | UI interactions |
| `evaluate_script` | Run JavaScript |
| `list_console_messages` | Get browser console output |
| `kill_app` / `hard_kill` | Stop running app |

## Screenshot Safety

All MCPs guard against 0x0 viewport. A 0x0 or sub-100-byte screenshot returns a text error — never sends a corrupt PNG to the AI.
