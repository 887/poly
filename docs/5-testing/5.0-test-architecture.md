## Overview

Three testing layers:

1. **Unit tests** — `cargo test` in each crate
2. **Mock test servers** — per-backend axum stubs for integration testing
3. **MCP devtools E2E** — AI-assisted visual testing via CDP/eval-bridge

## Test Servers

One mock server per backend, located in `servers/test-{backend}/`:

| Server | Port | Crate |
|---|---|---|
| `test-stoat` | dynamic | `poly-test-stoat` |

| `test-hackernews` | dynamic | `poly-test-hackernews` |

Plus `servers/test-common/` — shared `TestServerBase`, `/health`, `/reset`, `/seed`, `/reseed` helpers.

## Test Server Lifecycle Endpoints

Every test server implements:
- `POST /seed` — populate demo data (idempotent)
- `POST /reset` — wipe all data
- `POST /reseed` — wipe + reseed in one call
- `GET /health` — liveness check

## Test Auth Bypass

Each test server exposes `POST /test/auth/token`:
- Input: `{username}` (no password)
- Only works when URL contains `localhost` or `127.0.0.1`
- Returns a valid session token for testing

The UI uses this via `test_account_authenticate()` in `crates/core/src/ui/signup/mod.rs`.

## Screenshot Safety

All MCPs guard against 0x0 viewport screenshots in `mcp/devtools-protocol/src/mcp.rs`. A 0x0 or sub-100-byte image returns a text error instead of a corrupt PNG.
