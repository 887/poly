## Overview

Git forge backend surfacing repositories, Issues, Pull Requests, source code,
and notifications as Poly channels. The current implementation wraps the `gh`
CLI for GitHub / GitHub Enterprise, but the UI model (repos as servers, issues
and PRs as forum channels, code as a source explorer channel) is
forge-agnostic and designed to extend to **Gitea, Forgejo, GitLab**, or even
**local git repos** with a git-issue-style explorer.

**Crate:** `clients/github/` (`poly-github`)

## Data Model

| Forge Concept | Poly Concept |
|---|---|
| Repository | Server |
| Issue list | Forum channel (`ChannelType::Forum`) |
| PR list | Forum channel |
| Source tree | Code channel (`ChannelType::Code`) — file browser + reader |
| Issue/PR comments | Threaded messages under a post |
| Notifications | Notification feed |

## Channel Types

Each repo exposes three channels:

- **Issues** — forum-style issue list with threaded comment threads
- **Pull Requests** — forum-style PR list with threaded comment threads
- **Code** — source explorer with `list_files()` and `read_file()` for browsing the repo tree

## Authentication

Delegates to the user's `gh` CLI — no tokens are read from disk. All auth
flows go through `gh auth`. On WASM targets, the CLI is invoked via a
subprocess bridge on the native shell.

## Key Features

- Issue and PR browsing per repository
- Comment threads as threaded messages
- Source code file browser and reader (`ChannelType::Code`)
- Repos filtered to owner + collaborator with activity in last 2 years
- Read-only; write operations open the GitHub web UI
- GitHub Enterprise support via custom hostname

## Sibling: Forgejo Client

The `ChannelType::Code` and forum channel model is shared with the
**Forgejo client** (`clients/forgejo/`, doc [2.7b](2.7b-forgejo.md)),
which covers Forgejo, Gitea, and Codeberg via direct REST API.

Future forge extensions:

- **GitLab** — REST API v4, self-hosted or gitlab.com
- **Local git** — browse repos on disk, git-issue-style tracking
