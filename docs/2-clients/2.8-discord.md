# 2.8 Discord Client

> Status: In Progress (Phase 3.3)
> Sources: Discord API v10 official docs, twilight-rs, docs/archive/phases/phase-3.3-discord-plan.md
> Last updated: 2026-04-13

## Overview

Discord client for Poly. Connects to Discord's REST API v10 and Gateway WebSocket
for real-time events. Uses user tokens (direct API) with a self-hosted test server
for development.

**Crate:** `clients/discord/` (`poly-discord`)
**Feature flag:** `discord` / `dev-plugins`
**Test server:** `servers/test-discord/` on port 9102
**Test accounts:** Koala (🐨) + Kangaroo (🦘)

## Approach Decision

**Direct REST + Gateway WebSocket with user tokens.** This is the same approach used
by third-party Discord clients (Ripcord, BetterDiscord internals, etc.). It provides
full feature coverage but violates Discord's ToS — users are warned before connecting.

### Spacebar / Fosscord Support — First-Class Goal

[Spacebar](https://github.com/spacebarchat/server) (formerly Fosscord) is a
self-hostable, Discord-compatible backend that implements the Discord API v9/v10
surface. Because the user runs the server, there is **no ToS concern** — this
is how Poly is meant to be used long-term by power users.

**Implementation goals:**

1. **Spacebar is a first-class backend** — not an afterthought. Every feature we
   build for Discord must also work against a Spacebar instance pointed at via
   `base_url`.
2. **Dual-mode signup UI** — the account setup form offers two paths:
   - "Connect to Discord" (with the ToS warning)
   - "Connect to Spacebar instance" (prompts for custom `base_url`)
3. **Client explorer listing** — once mature enough, we aim for Poly to appear
   on <https://spacebar-explorer.sovr.top/clients> as a supported third-party
   client.
4. **Compatibility testing** — CI/manual smoke tests should include at least one
   run against a local Spacebar instance (possibly via `docker compose`), not
   just our in-house `servers/test-discord` mock.
5. **Version divergence tolerance** — Spacebar lags behind Discord's latest v10
   fields. Parse leniently (serde `#[serde(default)]`, `Option<T>` on new fields)
   so the client works against both.

## Discord API v10 — Key Concepts

### Authentication

Discord supports two auth modes:
- **Bot tokens:** `Authorization: Bot <token>` — requires registered application
- **User tokens:** `Authorization: <token>` (no prefix) — direct user access

Our client uses user tokens. The test server accepts both `Bot <token>` and
`Bearer <token>` prefixes for convenience.

### Snowflake IDs

All Discord IDs are snowflakes — 64-bit integers encoded as strings. They embed
a timestamp (ms since Discord Epoch: 2015-01-01T00:00:00.000Z), worker ID, process
ID, and increment. Snowflakes sort chronologically.

### Rate Limits

Discord enforces per-route and global rate limits. Response headers:
- `X-RateLimit-Remaining` — requests left in this window
- `X-RateLimit-Reset-After` — seconds until reset
- `X-RateLimit-Bucket` — shared bucket ID across related routes

On 429, respect `retry_after` in the JSON body. Our `DiscordHttpClient` should
parse these headers and back off automatically.

## Gateway WebSocket Protocol

The Gateway provides real-time events. Connection lifecycle:

### 1. Get Gateway URL

```
GET /api/v10/gateway
→ { "url": "wss://gateway.discord.gg" }
```

Connect to `{url}/?v=10&encoding=json`.

### 2. Hello (opcode 10) — received immediately

```json
{ "op": 10, "d": { "heartbeat_interval": 45000 } }
```

### 3. Start Heartbeating (opcode 1)

Send heartbeats every `heartbeat_interval` ms. First heartbeat after
`heartbeat_interval * jitter` (jitter ∈ [0, 1]). Payload: last sequence number
received (or `null` if none).

```json
{ "op": 1, "d": 251 }
```

Discord responds with Heartbeat ACK (opcode 11). If no ACK received before next
heartbeat, close connection and reconnect.

### 4. Identify (opcode 2)

```json
{
  "op": 2,
  "d": {
    "token": "user_token_here",
    "intents": 33281,
    "properties": { "os": "linux", "browser": "Poly", "device": "Poly" }
  }
}
```

**Intents for Poly** (bitfield):
- `GUILDS` (1 << 0) = 1 — guild create/update/delete, channel events
- `GUILD_MESSAGES` (1 << 9) = 512 — message events in guild channels
- `GUILD_MESSAGE_CONTENT` (1 << 15) = 32768 — message content (privileged for bots)
- `DIRECT_MESSAGES` (1 << 12) = 4096 — DM message events

Recommended intents value: `1 | 512 | 4096 | 32768` = **37377**

Additional optional intents:
- `GUILD_MEMBERS` (1 << 1) = 2 — member list (privileged)
- `GUILD_PRESENCES` (1 << 8) = 256 — presence updates (privileged)
- `GUILD_MESSAGE_TYPING` (1 << 11) = 2048 — typing indicators

### 5. Ready (opcode 0, t: "READY")

```json
{
  "op": 0, "s": 1, "t": "READY",
  "d": {
    "v": 10,
    "user": { "id": "...", "username": "...", ... },
    "guilds": [ { "id": "...", "unavailable": true }, ... ],
    "session_id": "...",
    "resume_gateway_url": "wss://gateway-us-east1-b.discord.gg"
  }
}
```

Cache `session_id` and `resume_gateway_url` for resuming.

### 6. Dispatch Events (opcode 0)

Key events for Poly:

| Event | Description | Poly Use |
|-------|-------------|----------|
| `MESSAGE_CREATE` | New message sent | Add to chat view |
| `MESSAGE_UPDATE` | Message edited | Update in chat view |
| `MESSAGE_DELETE` | Message deleted | Remove from chat view |
| `TYPING_START` | User typing in channel | Show typing indicator |
| `PRESENCE_UPDATE` | User status change | Update presence dots |
| `GUILD_CREATE` | Guild becomes available | Populate server list |
| `CHANNEL_CREATE` | New channel created | Update channel list |
| `CHANNEL_UPDATE` | Channel modified | Update channel list |
| `CHANNEL_DELETE` | Channel removed | Update channel list |

### 7. Resume (opcode 6) — after disconnect

Connect to `resume_gateway_url`, send:
```json
{
  "op": 6,
  "d": { "token": "...", "session_id": "...", "seq": 1337 }
}
```

### Gateway Opcodes Reference

| Code | Name | Direction | Description |
|------|------|-----------|-------------|
| 0 | Dispatch | Receive | Event dispatched (has `t` and `s`) |
| 1 | Heartbeat | Send/Receive | Keep-alive ping |
| 2 | Identify | Send | Initial handshake |
| 3 | Presence Update | Send | Update own status |
| 4 | Voice State Update | Send | Join/leave voice |
| 6 | Resume | Send | Resume dropped session |
| 7 | Reconnect | Receive | Must reconnect |
| 9 | Invalid Session | Receive | Session invalid |
| 10 | Hello | Receive | Connection established |
| 11 | Heartbeat ACK | Receive | Heartbeat acknowledged |

### Close Codes

| Code | Reconnect? | Meaning |
|------|-----------|---------|
| 4000 | Yes | Unknown error |
| 4001 | Yes | Unknown opcode |
| 4003 | Yes | Not authenticated |
| 4004 | **No** | Auth failed (bad token) |
| 4007 | Yes | Invalid seq (re-identify) |
| 4008 | Yes | Rate limited |
| 4009 | Yes | Session timed out |

## Data Model Mapping

### Discord → Poly Type Mapping

| Discord Type | Poly Type | Notes |
|-------------|-----------|-------|
| Guild | Server | `guild.name` → `server.name` |
| Channel (type 0/5) | Channel (Text) | Filter by type |
| Channel (type 1) | DmChannel | Via `/users/@me/channels` |
| Channel (type 3) | Group (DM) | Up to 10 participants |
| Channel (type 4) | Category | For channel tree structure |
| User | User | `global_name` preferred over `username` |
| Message | Message | ISO8601 timestamps |

### Avatar URLs

Discord avatar URLs follow this pattern:
```
https://cdn.discordapp.com/avatars/{user_id}/{avatar_hash}.png?size=128
```

For guilds:
```
https://cdn.discordapp.com/icons/{guild_id}/{icon_hash}.png?size=128
```

If avatar is null, use default:
```
https://cdn.discordapp.com/embed/avatars/{discriminator % 5}.png
```

## Current Implementation Status

### What's Done

- **`clients/discord/src/lib.rs`** — `DiscordClient` struct implementing `ClientBackend`
  - `authenticate()` — token auth via `GET /users/@me`
  - `get_servers()` — lists guilds via `GET /users/@me/guilds`
  - `get_channels()` — lists guild channels (filters type 0 + 5)
  - `get_messages()` — paginated message fetch
  - `send_message()` — text message send
  - `get_dm_channels()` — lists DM channels (filters type 1)
  - `get_user()` — fetch user by ID
- **`clients/discord/src/api.rs`** — API response types (DiscordUser, DiscordGuild, DiscordChannel, DiscordMessage)
- **`clients/discord/src/http.rs`** — HTTP client wrapper with token auth
- **`clients/discord/src/guest.rs`** — WASM guest stub (all methods return "not yet implemented")
- **`clients/discord/src/signup.rs`** — Token-based signup form UI
- **`servers/test-discord/`** — Mock REST server with seed data (Koala, Kangaroo, Wallaby)

### What's Missing

1. **Gateway WebSocket** — no real-time events yet (both client and test server)
2. **Avatar URL construction** — `avatar_url` always returns `None`
3. **Category support** — channels aren't grouped by category (`parent_id` ignored)
4. **Group DMs** — type 3 channels not handled
5. **Threads** — types 10/11/12 not handled
6. **Forum channels** — type 15 not handled
7. **Friends list** — `/users/@me/relationships` not called
8. **Typing indicators** — not sent or received
9. **Presence** — always returns `Offline`
10. **Message editing/deleting** — REST endpoints not implemented
11. **Reactions** — not implemented
12. **Embeds/attachments** — message content is text-only
13. **Rate limiting** — no backoff logic in HTTP client
14. **`global_name` field** — `DiscordUser` doesn't parse `global_name` (display name)
15. **WASM guest** — all methods are stubs
