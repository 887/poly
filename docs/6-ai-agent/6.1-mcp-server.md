## Overview

`poly-chat-mcp` — an MCP server that exposes every `ClientBackend` method as a tool. Runs embedded in the Poly app or standalone.

**Crate:** `mcp/chat-mcp/` (`poly-chat-mcp`)

## Architecture

```
Poly App (:3000 / :3001 / :3002)
  ├── ClientManager (shared backend pool)
  └── poly-chat-mcp (mounted as /mcp/* routes)
          ↓
poly-cli (connects via HTTP, discovers tools at runtime)
```

No `mcp.json` configuration needed for internal use — the MCP runs as HTTP routes inside the app. External MCP clients (Claude Code) can also connect via the same HTTP endpoint.

## Event Subscription Pattern (Phase C)

### Two delivery paths, same source

`poly-chat-mcp` exposes events two ways. Both read from the same
`EventStore` broadcast channel, so ordering is consistent across paths and
clients can mix-and-match.

| Path | Endpoint | Pattern | Compatible with |
|---|---|---|---|
| Polling | `POST /mcp` (`poll_events` tool) | Cursor-based pull | Every MCP host |
| Server-push | `GET /mcp/sse` | SSE stream of `notifications/event` JSON-RPC frames | MCP hosts that consume server-initiated frames (Continue.dev, Zed, Goose) |

The agent Integrations page in poly-web has a transport toggle persisted at
KV `agent.mcp.transport` (default `"polling"`); it changes which URL the
recommended config snippet shows. Both endpoints serve unconditionally — the
toggle is a recommendation, not a gate.

**Default is polling** because Claude Desktop's server-push consumer is
broken as of April 2026: it advertises `notifications/tools/list_changed`
and `notifications/prompts/list_changed` in the handshake but doesn't
actually refresh the client view ([anthropics/claude-code#4118](https://github.com/anthropics/claude-code/issues/4118),
closed-as-duplicate of #13646). Flip the toggle to `"sse"` when the upstream
fix lands or when targeting a host that handles server-push reliably.

### The two tools

```
subscribe_events({
  account_ids?: [string],
  chat_ids?:    [string],
  event_types?: [string],
}) → { subscription_id: string }

unsubscribe_events({ subscription_id }) → { ok: true }

poll_events({
  since_ms:        number,             // wall-clock cursor
  limit?:          number,             // ≤ 500 (server-capped)
  account_ids?:    [string],
  chat_ids?:       [string],
  event_types?:    [string],
  subscription_id?: string,
}) → {
  events:        [Event],
  next_since_ms: number,               // pass back as since_ms next call
}
```

`subscribe_events` reserves a filter set + `subscription_id` so subsequent
`poll_events` can be a one-arg lookup. Calling `poll_events` without any
subscription works too — the filter args are then per-call.

### Delivery internals

`mcp/chat-mcp/src/events.rs::EventStore` owns a `tokio::sync::broadcast`
channel **per account**. `BackendPool::insert(account_id, backend)`
spawns a fan-out task that reads `backend.event_stream()` (the existing
`Stream<Item = ClientEvent>` from the `ClientBackend` trait) and forwards
each event into the broadcast channel. `BackendPool::remove(account_id)`
cancels the task.

The broadcast channel is a **ring buffer**:
- capacity 2000 events
- 5-minute TTL on retained events

`poll_events` reads from the ring, applies the filter, and clamps the
result at 500 events / call. `next_since_ms` is the wall-clock of the
last event returned (or the request timestamp if the result was empty)
so the client can advance the cursor cheaply.

### Event shapes

The events that flow through are the existing `ClientEvent` variants from
`clients/client/src/types.rs` — `MessageReceived`, `MessageEdited`,
`MessageDeleted`, `TypingStarted`, `PresenceChanged`,
`ChannelUpdated`, `ServerUpdated`, `FriendRequestReceived`,
`ConnectionStateChanged`, `VoiceUserJoined/Left`, `VoiceStateUpdated`,
`SidebarInvalidated`, `NotificationReceived`. JSON-serialized 1:1.

### Recommended cadence

For an active Claude Desktop session driving 10 concurrent chats:
- `poll_events({since_ms, limit: 500})` every 2–5 seconds is the sweet
  spot — fast enough to feel reactive, slow enough that an idle session
  costs almost nothing.
- For background heartbeat (per `plan-meta-personalities.md`), 60s is
  fine.

### Verifying it works

```bash
# In one terminal: subscribe + poll loop (paste into Claude Desktop or use poly-cli)
poly-cli call subscribe_events --account_ids '["disc-koala"]'
# → { subscription_id: "sub-abc123" }

poly-cli call poll_events --subscription_id sub-abc123 --since_ms 0
# → { events: [...], next_since_ms: 1716239012345 }

# In another terminal: trigger an event
curl -X POST http://localhost:9102/test/discord/send_message -d '...'

# Re-poll: the new event appears
poly-cli call poll_events --subscription_id sub-abc123 --since_ms 1716239012345
```

Integration test: `phase_c_discord_message_received_via_poll_events`
(in `mcp/chat-mcp/tests/`).

### When to flip to SSE

The `agent.mcp.transport` toggle defaults to `"polling"`. Flip to `"sse"`
when **any** of:

1. [anthropics/claude-code#4118](https://github.com/anthropics/claude-code/issues/4118)
   closes (Claude Desktop notifications actually refresh the client view), AND
   Claude Desktop release notes document a stable `notifications` capability
   flag in the handshake.
2. The MCP host you're using is **not** Claude Desktop — Continue.dev, Zed,
   Goose, OpenClaude all handle server-push reliably today.
3. You're integration-testing the SSE path (e.g. `curl -N
   http://127.0.0.1:3010/mcp/sse` and watching frames stream).

The Poly server doesn't need a restart to honor the flip — both endpoints
are always live. Only the recommended snippet in the Integrations page
changes.

### Implementation pointers

- SSE handler: `mcp/chat-mcp/src/main.rs::handle_mcp_sse` — uses
  `axum::response::sse::{Sse, Event, KeepAlive}` and
  `tokio_stream::wrappers::BroadcastStream` to wrap each `McpEvent` from
  the broadcast channel as a `notifications/event` JSON-RPC frame.
- Lagged-receiver behavior: a slow SSE consumer that overruns the 2000-event
  ring drops the missed frames silently. Hosts that need gap detection can
  fall back to `poll_events` with their last-seen `since_ms` cursor.
- Keep-alive: default axum SSE keep-alive (15s comment frame) is enough to
  stop intermediate proxies / browsers from closing idle connections.
