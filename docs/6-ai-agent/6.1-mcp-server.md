## Overview

`poly-chat-mcp` — an MCP server that exposes every `ClientBackend` method as a tool. Runs embedded in the Poly app or standalone.

**Crate:** `mcp/chat-mcp/` (`poly-chat-mcp`)

## Architecture

```
Poly App (:3000 / :3001 / :3002)
  ├── ClientManager (shared backend pool)
  └── poly-chat-mcp (mounted as /mcp/* routes)
          ↓
poly-cli (connects via HTTP, discovers tools at runtime)
```

No `mcp.json` configuration needed for internal use — the MCP runs as HTTP routes inside the app. External MCP clients (Claude Code) can also connect via the same HTTP endpoint.

## `test_signin` Tool

- Input: `{backend, url, username}` — no password
- Guard: only works when `url` contains `localhost` or `127.0.0.1`
- Calls `/test/auth/token` → logs in with returned token
- Available for: stoat, matrix, lemmy, discord, teams test servers

## Event Subscription Pattern (Phase C)

### Why polling, not push

MCP spec `2025-11-25` introduced the **Streamable HTTP** transport with proper
server-push (SSE streams on GET / response-streamed POST). The MCP layer is
ready for push delivery.

Claude Desktop's consumer side, however, is incomplete as of 2026-05:
`notifications/tools/list_changed` and `notifications/prompts/list_changed`
are advertised in the handshake but don't actually refresh the client view
in practice ([anthropics/claude-code#13646](https://github.com/anthropics/claude-code/issues/13646),
April 2026). Other server-initiated frame types are not confirmed
supported either.

**Decision:** ship a polling-based delivery path that any MCP host can
consume. Re-open SSE/streaming when (a) Claude Desktop closes the
list_changed bug AND (b) a non-Desktop host emerges as the primary
target for poly-chat-mcp users.

### The two tools

```
subscribe_events({
  account_ids?: [string],
  chat_ids?:    [string],
  event_types?: [string],
}) → { subscription_id: string }

unsubscribe_events({ subscription_id }) → { ok: true }

poll_events({
  since_ms:        number,             // wall-clock cursor
  limit?:          number,             // ≤ 500 (server-capped)
  account_ids?:    [string],
  chat_ids?:       [string],
  event_types?:    [string],
  subscription_id?: string,
}) → {
  events:        [Event],
  next_since_ms: number,               // pass back as since_ms next call
}
```

`subscribe_events` reserves a filter set + `subscription_id` so subsequent
`poll_events` can be a one-arg lookup. Calling `poll_events` without any
subscription works too — the filter args are then per-call.

### Delivery internals

`mcp/chat-mcp/src/events.rs::EventStore` owns a `tokio::sync::broadcast`
channel **per account**. `BackendPool::insert(account_id, backend)`
spawns a fan-out task that reads `backend.event_stream()` (the existing
`Stream<Item = ClientEvent>` from the `ClientBackend` trait) and forwards
each event into the broadcast channel. `BackendPool::remove(account_id)`
cancels the task.

The broadcast channel is a **ring buffer**:
- capacity 2000 events
- 5-minute TTL on retained events

`poll_events` reads from the ring, applies the filter, and clamps the
result at 500 events / call. `next_since_ms` is the wall-clock of the
last event returned (or the request timestamp if the result was empty)
so the client can advance the cursor cheaply.

### Event shapes

The events that flow through are the existing `ClientEvent` variants from
`clients/client/src/types.rs` — `MessageReceived`, `MessageEdited`,
`MessageDeleted`, `TypingStarted`, `PresenceChanged`,
`ChannelUpdated`, `ServerUpdated`, `FriendRequestReceived`,
`ConnectionStateChanged`, `VoiceUserJoined/Left`, `VoiceStateUpdated`,
`SidebarInvalidated`, `NotificationReceived`. JSON-serialized 1:1.

### Recommended cadence

For an active Claude Desktop session driving 10 concurrent chats:
- `poll_events({since_ms, limit: 500})` every 2–5 seconds is the sweet
  spot — fast enough to feel reactive, slow enough that an idle session
  costs almost nothing.
- For background heartbeat (per `plan-meta-personalities.md`), 60s is
  fine.

### Verifying it works

```bash
# In one terminal: subscribe + poll loop (paste into Claude Desktop or use poly-cli)
poly-cli call subscribe_events --account_ids '["disc-koala"]'
# → { subscription_id: "sub-abc123" }

poly-cli call poll_events --subscription_id sub-abc123 --since_ms 0
# → { events: [...], next_since_ms: 1716239012345 }

# In another terminal: trigger an event
curl -X POST http://localhost:9102/test/discord/send_message -d '...'

# Re-poll: the new event appears
poly-cli call poll_events --subscription_id sub-abc123 --since_ms 1716239012345
```

Integration test: `phase_c_discord_message_received_via_poll_events`
(in `mcp/chat-mcp/tests/`).

### Future: SSE re-open conditions

Re-evaluate when:
1. `anthropics/claude-code#13646` closes (list_changed actually
   refreshes), AND
2. Claude Desktop release notes document a `notifications` or
   `streaming` capability flag in the handshake, OR
3. A non-Desktop MCP host (Continue.dev, Zed, Goose, OpenClaude) becomes
   the primary target for poly-chat-mcp users.

When re-opened: the Streamable HTTP transport supports SSE on the same
HTTP route — implementation would extend `mcp/chat-mcp/src/server.rs` to
keep the response stream open and write JSON-RPC notification frames as
events arrive on the broadcast channel. The `EventStore` ring buffer
stays as-is (it's the same source).
