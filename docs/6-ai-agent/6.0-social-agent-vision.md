## Core Concept

Every logged-in account across all backends becomes a set of MCP tools. An AI agent (Claude, or any MCP-compatible model) can:

1. **See** all incoming messages across all accounts in real-time
2. **Draft** responses with per-chat personality, mood, and memory context
3. **Send** responses after human approval (or auto-send if configured)
4. **Learn** from the user's actual responses to improve future drafts
5. **Schedule** proactive outreach ("message X about Y every N days")
6. **Simulate** realistic typing behavior (delays, typing indicators)

The agent shares the same live connections as the Poly UI — no duplicate logins, no separate sessions.

## Architecture

```
Poly App (UI)
  ├── Matrix, Stoat, Discord, Teams, Poly accounts
  └── Shared ClientBackend Pool
          ↓
  Social Agent MCP Server
  ├── /tools/messages/*    (read, search, history)
  ├── /tools/send/*        (draft, approve, send)
  ├── /tools/contacts/*    (list, profile, memory)
  ├── /tools/schedule/*    (outreach plans, triggers)
  ├── /tools/personality/* (per-chat persona)
  └── /events/incoming     (message stream → AI)
```

## Modes

- **Embedded** — runs inside Poly app (same process), shares the backend pool
- **Standalone** — separate binary connecting to backend pool via IPC/socket
- **Headless** — no UI, pure MCP + REST

## Phase Split

- **Phase 5.1** (IN-PROGRESS) — MCP Backend + CLI Tool + E2E testing. See [6.1-mcp-server.md](6.1-mcp-server.md)
- **Phase 5.2** (TBD) — Personality engine, memory/learning, auto-response pipeline, typing simulation, AI provider integration

## Open Architecture Decisions

- Process model: embedded vs. standalone vs. both
- MCP transport: stdio vs. SSE vs. both
- AI provider abstraction: pluggable trait (Claude, GPT, Gemini, Ollama)
- Approval UX: in-app notification, desktop notification, or terminal prompt
- Storage: per-chat memories, personality configs, outreach schedules
