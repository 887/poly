## Architecture Pattern (all platforms)

1. `dx serve --platform web --port <PORT>` compiles the app as WASM
2. A thin native shell (Chrome / Wry / Electron) loads from the dev server
3. On code changes, only the WASM reloads — the native window stays alive
4. MCP reconnects via CDP or eval-bridge after each rebuild

## Fullstack Architecture

All three UI apps are Dioxus **fullstack** apps. `dx serve --fullstack` builds:
- WASM client bundle (runs in browser/webview)
- Native axum server (runs natively, merges `poly_host::router()`)

Both halves come from the same `src/main.rs`. The server half serves the WASM bundle and also handles `/host/*` routes — one process, one port.

## Host Bridge

The WASM client communicates with the native layer via `POST /host/*` HTTP calls to the same-process server. See [1.2-host-bridge.md](../1-architecture/1.2-host-bridge.md) for details.

## MCP Devtools

Each platform has a dedicated MCP devtools server for AI-assisted development:
- `mcp/web-devtools-mcp/` — manages Chrome CDP on port 9222
- `mcp/electron-devtools-mcp/` — manages Electron CDP on port 9224
- `mcp/desktop-devtools-mcp/` — manages Wry eval-bridge on port 9223

MCP workflow: `launch_app` → poll `get_last_build_status` → `connect_cdp` → `take_screenshot` / `navigate`

See [5.2-mcp-devtools.md](../5-testing/5.2-mcp-devtools.md) for details.
