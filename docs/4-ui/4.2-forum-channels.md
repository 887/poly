> Status: Draft

## 1. Overview

Add a `Forum` channel type to Poly's unified messenger. Forum channels display a
list of posts (not a chat scroll) with sort/filter controls, vote buttons, and
threaded comment views. This supports Discord forum channels, Lemmy communities,
Reddit-style subreddits, and Hacker News-style boards through a single unified UI.

### What changes

| Layer | Summary |
|-------|---------|
| **Types** (`clients/client/src/types.rs`) | New `ChannelType::Forum`, `ForumPost`, `ForumSortOrder`, `ForumFilter`, `ForumTag`, `ThreadComment`, `VoteDirection`; add vote fields to `Message` |
| **WIT** (`wit/messenger-plugin.wit`) | Mirror all new types and trait methods in the WIT interface |
| **ClientBackend trait** (`clients/client/src/lib.rs`) | 5 new default methods for forum operations |
| **Events** (`clients/client/src/events.rs`) | New event variants for forum post/comment/vote changes |
| **Plugin bridge** (`crates/plugin-host/src/bridge.rs`) | Bridge new WIT exports to `ClientBackend` trait methods |
| **Demo backend** (`clients/demo/src/data.rs`) | Add a forum channel with demo posts, comments, tags |
| **UI components** (`crates/core/src/ui/`) | `ForumPostList`, `ForumPostCard`, `ForumSortBar`, `ThreadedCommentView`, `VoteButtons` |
| **Routing** (`crates/core/src/ui/routes.rs`) | Content area switches on `ChannelType::Forum`; add post-detail route |
| **Sidebar** (`crates/core/src/ui/account/common/channel_list.rs`) | Forum icon in channel list |
| **CSS** | New stylesheet for forum layout, post cards, vote buttons, comment tree |

/// A post in a forum channel.
#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
pub struct ForumPost {
    /// Backend-specific post ID.

    pub vote_score: Option<i32>,
    /// The current user's vote on this post. `None` if not voted or not supported.
    pub user_vote: Option<VoteDirection>,
    /// Number of comments/replies.
    pub comment_count: u32,
    /// Whether this post is pinned to the top.
    pub pinned: bool,
    /// Whether this post is locked (no new comments).
    pub locked: bool,
    /// When the post was created.
    pub created_at: DateTime<Utc>,
    /// When the last comment or edit occurred.
    pub last_activity: DateTime<Utc>,
    /// Backend-specific channel this post belongs to.
    pub channel_id: String,
}

### 2.7 `ThreadComment`

```rust
/// A comment in a threaded post discussion.
///
/// Comments form a tree: each comment has an optional `parent_id` pointing
/// to its parent comment. Top-level comments have `parent_id == None`.
#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
pub struct ThreadComment {
    /// Backend-specific comment ID.
    pub id: String,
    /// Comment author.
    pub author: User,
    /// Comment content.
    pub content: MessageContent,
    /// When the comment was posted.
    pub timestamp: DateTime<Utc>,
    /// Nesting depth (0 = top-level reply to the post).
    pub depth: u32,
    /// Net vote score. `None` for backends without voting.
    pub vote_score: Option<i32>,
    /// The current user's vote on this comment. `None` if not voted.
    pub user_vote: Option<VoteDirection>,
    /// Number of direct child comments.
    pub children_count: u32,
    /// Parent comment ID. `None` for top-level comments.
    pub parent_id: Option<String>,
    /// Whether the comment has been edited.
    pub edited: bool,
    /// The post this comment belongs to.
    pub post_id: String,
}
```

    Err(ClientError::NotSupported("create_forum_post".to_string()))

### 9.1 `ChatData` Additions

Add forum-specific state to `ChatData` (or a new `ForumData` struct stored
alongside):

```rust
/// Forum state for the currently viewed forum channel.
pub struct ForumState {
    /// Currently loaded posts.
    pub posts: Vec<ForumPost>,
    /// Current filter/sort state.

    /// Whether more posts are available (pagination).

    /// Whether posts are currently loading.
    pub loading: bool,
}

/// Thread view state for the currently viewed post.
pub struct ThreadState {
    /// The post being viewed.
    pub post: Option<ForumPost>,
    /// Loaded comments (flat list, tree built in UI).
    pub comments: Vec<ThreadComment>,
    /// Current comment sort order.
    pub comment_sort: ForumSortOrder,
    /// Whether comments are currently loading.

```

Store as `Option<ForumState>` and `Option<ThreadState>` in `ChatData`. Set to
`None` when the user navigates away from a forum channel. This avoids polluting
the existing `messages` / `current_channel` state used by chat views.

### 9.2 Optimistic Vote Updates

Vote actions use optimistic updates:

1. Immediately update `vote_score` and `user_vote` in the local `ForumState` or `ThreadState`
2. Fire `backend.vote(target, direction)` asynchronously
3. On error, revert the local state and show a toast notification

## 13. Backend Mapping

How forum channels map to each backend:

| Backend | Forum Source | Vote Support | Thread Model |
|---------|-------------|--------------|--------------|
| Discord | Forum channels (type 15) | No (reactions only) | Discord threads (each post is a thread) |
| Matrix  | Spaces with threaded rooms | No (reactions only) | Matrix threads (MSC3440) |
| Stoat   | TBD (Revolt has no forums) | No | N/A |
| Teams   | N/A (no forum equivalent) | No | N/A |
| Lemmy   | Communities | Yes (upvote/downvote) | Nested comment tree |
| Poly    | Native forum channels | Yes | Native threaded comments |
| Demo    | Static demo data | Yes (in-memory) | Static nested comments |

Backends that don't support forums should:
- Never return `ChannelType::Forum` channels from `get_channels()`
- Return `NotSupported` from all forum methods (the default trait implementations)

## 15. Open Questions

1. **Markdown rendering:** Should forum post bodies support full markdown? The
   current chat view renders plain text. If yes, we need a markdown-to-HTML
   renderer (pulldown-cmark compiled to WASM, or render markdown server-side).

2. **Image previews in cards:** Should the post card show inline image previews
   from attachments, or only the explicit `thumbnail_url`?

3. **Flat vs threaded toggle:** Some users prefer Reddit-style nested threads,
   others prefer Discord-style flat chronological threads. Should we support both
   via a toggle, or commit to one style per backend?

4. **Comment editing:** Should comments be editable after posting? If yes, need
   to add an edit flow similar to message editing.

5. **Post deletion:** Who can delete posts? Only the author, or also moderators?
   This depends on the permission system (not yet implemented).

6. **Cross-post detection:** Should the UI detect when the same content is posted
   to multiple forum channels (common in Lemmy/Reddit) and show a
   "cross-posted from" indicator?
