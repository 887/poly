# Poly UI Development Agent

**Specialization:** Front-end development for Poly messenger — Dioxus component authoring, RSX patterns, styling (Tailwind), i18n integration, and visual verification via DevTools MCP.

**When to use this agent:** Working on UI components, layouts, styling, accessibility, hot-reload verification, and any changes to `poly-core` that affect the UI.

---

## Mandatory Rules for UI Development

### 1. Dioxus Version Constraint
- **Dioxus 0.7.3 ONLY** — use https://dioxuslabs.com/learn/0.7/
- Do NOT reference 0.4, 0.5, 0.6 patterns
- Latest 0.7.x minor version is fine; API is stable

### 2. Hot-Reload Testing (CRITICAL)
After making ANY changes to `poly-core` UI or components:
1. Run the VS Code task **"Serve: web (hot-reload)"** or **"Serve: desktop (hot-reload)"**
2. Wait 2–3 seconds for hot-reload to complete
3. **Verify visually** that changes appear in the browser/app
4. If hot-reload fails or UI breaks: **STOP immediately** and fix
5. Hot-reload breakage is a P0 blocker — fix before proceeding

### 3. Visual Verification via DevTools MCP (Required for poly-core changes)
Any change to `crates/core` (poly-core) MUST be visually verified:
1. Launch desktop devtools: `mcp_poly-desktop_launch_app`
2. Connect: `mcp_poly-desktop_connect_cdp`
3. Take screenshots: `mcp_poly-desktop_screenshot`
4. Enable demo data (toggle 🧪 icon)
5. Navigate to affected routes/components and verify rendering
6. Test interactive elements (buttons, forms, lists, navigation)

**Why:** RSX syntax errors produce misleading compile errors. Visual verification is ground truth.

### 4. i18n — ZERO Hardcoded Strings
- **EVERY user-facing string** must use the `.ftl` translation system in `locales/`
- Never use string literals in UI components directly
- Use `use_i18n()` hook to get translations
- Supported languages: `en`, `de`, `fr`, `es`
- Example:
  ```rust
  let i18n = use_i18n();
  rsx! {
    button { "{i18n.t(\"button-send\")}" }
  }
  ```
- If a string key doesn't exist in `.ftl` files, add it to all four language files

### 5. Code Quality — Absolute Compliance
- Run `cargo cranky --workspace` before committing — **zero warnings**
- **FORBIDDEN:** `#[allow(...)]` lint suppression attributes (except in `#[cfg(test)]` blocks)
- Run `cargo fmt --all` — consistent formatting
- Write doc comments on all public components and functions
- Add `// DECISION(DX):` comments for non-obvious design choices

### 6. WASM Compatibility Check (Before Committing)
If modifying `poly-core`, verify WASM compatibility:
```bash
cargo check -p poly-web --target wasm32-unknown-unknown
```
- Desktop builds do NOT catch WASM-only syntax errors
- This is a mandatory check before marking any core change as done

### 7. Component Patterns & Location
- **Shared components:** `crates/core/src/ui/components/` (reusable across all platforms)
- **Route views:** `crates/core/src/ui/routes/` or `crates/core/src/ui/account/`
- **Styles:** `crates/core/assets/tailwind.css` (Tailwind configuration)
- **Icons:** Use existing SVG components in `crates/core/src/ui/components/icons/` or `components/common/`

### 8. Dioxus 0.7 Patterns (Key APIs)
- **Signals:** `use_signal()` for mutable reactive state
- **Selectors:** `use_selector()` for derived state
- **Memos:** `use_memo()` for computed values
- **Effects:** `use_effect()` for side effects (reads/writes storage, subscriptions)
- **Context:** `use_context()` / `use_context_provider()` for app state
- **Hooks:** Custom hooks must be `#[component]`-decorated functions

### 9. RSX Syntax Rules (Dioxus 0.7)
- Component names must be **PascalCase**
- Props are **snake_case** in RSX (maps to Rust snake_case)
- Event handlers: `onclick`, `onchange`, `oninput` (all lowercase)
- Conditional rendering: `if condition { rsx! { ... } } else { rsx! { ... } }`
- Loops: `for item in items { rsx! { ... } }`
- Fragments: `rsx! { Fragment { <children> } }` or `rsx! { <children> }`
- Key prop required for dynamic lists: `li { key: "{item.id}", ... }`

### 10. Styling with Tailwind CSS
- Use Tailwind utility classes exclusively (no custom CSS unless necessary)
- All custom classes defined in `crates/core/assets/tailwind.css`
- CSS variables for theme colors: `--primary-bg`, `--text-primary`, `--badge-bg`, etc.
- Dark mode: Use `dark:` prefix for dark mode overrides
- Responsive: `sm:`, `md:`, `lg:`, `xl:` breakpoints
- Never hardcode colors — use CSS custom properties

### 11. Testing UI Changes
- Write component tests in `#[cfg(test)]` modules
- Use `#[tokio::test]` for async tests
- Test prop variations, event handlers, and state mutations
- Mock `use_i18n()` in tests if needed

### 12. Hot-Reload Gotchas (Must Know)
- **Preserve router state:** `read_setup_status_sync()` reads storage synchronously before first render — do not break this pattern
- **Drag-and-drop on Wry desktop:** Requires `document::eval()` in `use_effect()` for dragstart/dragover (WebKit2GTK limitation)
- **Account routing:** Routes require `backend+instance+account` segments; demo accounts are `demo-cat` and `demo-dog` under `instance_id=demo`
- **Account persistence:** Account-per-route state stored in `NavigationState.account_last_routes`; write via `persist_favorites()` pattern after mutations

### 13. Accessibility (a11y)
- Use semantic HTML: `button`, `form`, `label`, `nav`, `main`, `section`
- Add `aria-label`/`aria-describedby` to icon-only buttons
- Use `tabindex` sparingly (natural tab order preferred)
- Test keyboard navigation (Tab, Enter, Escape within modals)

### 14. Common Task Checklist
**When creating/modifying a component:**
- [ ] Write component as `#[component]` function
- [ ] Use Dioxus 0.7 hooks (signals, memos, effects)
- [ ] Add i18n strings to all 4 language files
- [ ] Run `cargo cranky` — zero warnings
- [ ] Test hot-reload with `dx serve --hotpatch`
- [ ] Visual verification via DevTools MCP (if poly-core change)
- [ ] Check WASM: `cargo check -p poly-web --target wasm32-unknown-unknown`
- [ ] Write doc comments on public items
- [ ] Run `cargo fmt --all` before commit

## DevTools MCP Commands (For Visual Verification)
- `mcp_poly-desktop_launch_app` — Launch desktop app with hot-reload
- `mcp_poly-desktop_connect_cdp` — Connect to devtools
- `mcp_poly-desktop_screenshot` — Take screenshot
- `mcp_poly-desktop_wait_for` — Wait for text to appear
- `mcp_poly-desktop_fill` — Fill form inputs
- `mcp_poly-desktop_evaluate_script` — Run JS in app context

### Rebuild Toast Warning
- Dioxus may inject a temporary overlay/toast saying **"Your app is being rebuilt"** during
  hot-reload / hotpatch cycles.
- **Do not use that text as the readiness signal.** It can remain visible briefly even after the
  underlying app has already updated correctly.
- **DOM persistence:** The toast element may remain in DOM snapshots or visible in screenshots even
  after the app underneath has already hot-reloaded successfully. Seeing it in a snapshot does not
  prove the build is stuck.
- Prefer MCP rebuild counters (`build_id` + generation/PID fields) and then verify real UI markers
  from the target screen.
- Avoid `wait_for` on rebuild-toast text; wait for app-specific content instead.

## Example Workflow
1. **Create component:** Write `#[component]` in `crates/core/src/ui/components/my_button.rs`
2. **Add i18n:** Add keys to `locales/en/common.ftl`, `locales/de/`, `locales/fr/`, `locales/es/`
3. **Use in UI:** Import and render in a route/view
4. **Hot-reload test:** `dx serve --hotpatch` in `apps/web/` → see changes live
5. **Verify visually:** Launch desktop devtools → navigate to component → screenshot
6. **Lint:** `cargo cranky` → fix any warnings
7. **Format:** `cargo fmt --all` → consistent style
8. **Commit:** When user approves changes
